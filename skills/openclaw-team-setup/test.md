# OpenClaw 多智能体联调测试

用于验证以下能力是否生效：

- 飞书 `pm / dev / qa` 是否分别命中对应智能体
- `pm -> qa` 原生 sub-agent 是否可用
- `pm -> dev` 协作链路是否可用
- `pm -> dev + qa` 完整协作是否可用

## 第一轮：角色路由测试

目标：确认三个飞书机器人分别进入正确智能体，而不是全部落到 `main`。

发给 `pm` 机器人：

```text
请只回复这六个字符：ROLE=pm
```

发给 `dev` 机器人：

```text
请只回复这八个字符：ROLE=dev
```

发给 `qa` 机器人：

```text
请只回复这七个字符：ROLE=qa
```

预期结果：

- `pm` 只回 `ROLE=pm`
- `dev` 只回 `ROLE=dev`
- `qa` 只回 `ROLE=qa`

如果三个机器人回复完全一样，或者都表现得像同一个人格，说明路由仍有问题。

## 第二轮：只测 pm -> qa

目标：确认 `pm` 会调用 `qa` 做测试分析，而不是自己伪装输出。

发给 `pm` 机器人：

```text
不要调用 dev。
只调用 qa 做一次测试分析。
任务：为“登录按钮无响应”设计测试点。
最终回复必须分成两段：
第一段标题写【pm】
第二段标题写【qa】
```

预期结果：

- 回复中明确包含 `【pm】`
- 回复中明确包含 `【qa】`
- `【qa】` 区块内容应偏测试分析，例如：
  - 测试范围
  - 测试点
  - 边界场景
  - 验证结论

如果没有 `【qa】` 区块，通常说明 `pm` 没真正派发到 `qa`。

## 第三轮：只测 pm -> dev

目标：确认 `pm` 会调用开发链路，而不是自己直接回答。

发给 `pm` 机器人：

```text
不要调用 qa。
只调用 dev 做一次开发分析。
任务：分析“登录按钮无响应”的可能根因和修复思路。
最终回复必须分成两段：
第一段标题写【pm】
第二段标题写【dev】
```

预期结果：

- 回复中明确包含 `【pm】`
- 回复中明确包含 `【dev】`
- `【dev】` 区块内容应偏开发分析，例如：
  - 根因分析
  - 实现方案
  - 改动点
  - 风险说明

如果没有 `【dev】` 区块，说明 `pm` 没真正走开发协作链路。

## 第四轮：完整协作测试

目标：验证 `pm / dev / qa` 的完整协同输出。

发给 `pm` 机器人：

```text
按团队协作处理这个任务：
修复“登录按钮无响应”。
要求：
1. pm 先拆解任务
2. dev 给开发方案
3. qa 给测试点
4. 最终按【pm】【dev】【qa】三个区块回复
```

预期结果：

- 回复包含 `【pm】`
- 回复包含 `【dev】`
- 回复包含 `【qa】`
- 三部分内容风格明显不同
- `pm` 像协调者
- `dev` 像开发者
- `qa` 像测试者

## 失败时怎么判断

现象：三个机器人回复风格相同

可能原因：

- 飞书路由没有命中对应 agent
- Gateway 仍在使用旧配置

现象：`pm` 回复里没有 `【qa】` 或 `【dev】`

可能原因：

- `pm` 没有真正调用子智能体
- `pm` 只是自己模拟输出

现象：`dev` 或 `qa` 单独测试正常，但 `pm` 协作失败

可能原因：

- `pm` 提示词没有触发真实委派
- `sub-agent` 或 ACP 链路没被真正调用

## 建议

- 每轮测试保留原始回复
- 如果某一轮失败，记录测试时间
- 失败后把原始回复发出来，便于对照 `gateway.log` 精确定位
