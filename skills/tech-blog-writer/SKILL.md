# Tech Blog Writer

这个 skill 用来把一张“选题卡片”扩写成一个可继续加工的博客文件夹，而不是只输出一段正文。

它最适合接在 `ai-topic-research` 后面使用：

- `ai-topic-research` 负责选题、给支撑链接和写作建议
- 再补一轮主题搜索，整理出可用的素材资料
- 本 skill 负责把这些输入整理成一个博客目录，目录里包含正文和配图要求

## 适用场景

- 用户已经有选题，不想从空白页开始写
- 用户已经有几条支撑链接，希望整理成一篇博客
- 用户希望文章自带配图占位符，后续再补图
- 用户想写技术博客，而不是公众号排版稿或营销软文

## 输入要求

正常情况下，尽量具备这三类输入：

- `选择主题`
- `素材资料`
- `写作建议`

如果还有 `支撑链接`，更适合把它作为“补素材”阶段的起点，而不是直接拿来写正文。

如果缺一部分，也可以继续写，但要优先保留已有信息，不要编造事实。

## 工作流程

1. 先理解主题和读者
2. 从素材资料里提炼文章主线和论据
3. 把写作建议转成标题、开头、结构和收束方式
4. 输出博客文件夹
5. 生成 `blog.md`
6. 生成 `image-requirements.md`
7. 为整篇文章规划 1 张封面图和若干正文配图
8. 在 `blog.md` 合适位置插入配图占位符

## 配图风格预设

生成 `image-requirements.md` 时必须先选择一个统一配图风格。默认使用 `清新手绘风`，除非用户明确指定其他风格。

可用预设：

- `清新手绘风`（默认）：浅色背景、柔和蓝绿/薄荷/米白配色、手绘线条、轻量水彩质感、圆角卡片、少量拟物小图标，适合公众号技术文章，亲和但不幼稚
- `科技信息图风`：深色或渐变背景、发光连线、节点图、仪表盘、科技蓝/青色，适合架构、协议、系统工程主题
- `极简线框风`：白底、细线、少量强调色、结构清楚，适合教程、步骤、概念解释
- `杂志拼贴风`：卡片拼贴、轻微纸张纹理、截图式模块、醒目的标题区，适合趋势判断、工具盘点
- `产品界面风`：接近真实应用 UI 的面板、列表、状态卡、命令行窗口，适合工具使用方法和 workflow 展示

要求：

- `image-requirements.md` 的“整体要求”必须写明选用的风格预设
- 同一篇文章的封面图和正文图必须使用同一个风格预设
- 默认不要用深色赛博风，除非主题确实需要
- 默认不要让图片里出现大量文字，避免生成不准或公众号端阅读吃力
- 对技术工具教程类文章，优先用“清新手绘风 + 轻量流程图/卡片图”的组合
- 配图不能只做氛围装饰，必须至少有 `1` 张高信息密度图，能被读者截图收藏
- 高信息密度图优先表达：`工具/概念 -> 能力 -> 使用场景 -> 第一步行动`
- 即使是清新手绘风，也要把图做成“可读的信息卡片 / 流程图 / 对照图”，而不是只有机器人、云朵、卡片这类泛化元素

## 默认输出目录

正常情况下，输出一个目录，路径约定为：

```text
tech-blog-writer/YYYYMMDD/<topic-slug>/
  blog.md
  image-requirements.md
  assets/
```

例如：

- `tech-blog-writer/20260410/rag-long-context-memory/`
- `tech-blog-writer/20260410/agent-protocols/`
- `tech-blog-writer/20260410/ai-coding-agents/`

### `blog.md`

这是正文文件，正常情况下包含：

1. 标题
2. 导语
3. 正文
4. 结尾

正文内部建议包含：

- 为什么这个主题现在值得写
- 它解决了什么问题
- 社区现在在怎么讨论它
- 读者应该怎么判断要不要跟进
- 读者看完以后第一步应该怎么做

### `image-requirements.md`

这是配图需求文件，正常情况下每张图都要写清楚：

- 整体风格要求
- 封面图要求
- 图片编号
- 建议放置位置
- 图片用途
- 画面内容
- 风格建议
- 尺寸 / 比例建议

建议格式像这样：

```md
## 整体要求

- 风格预设：清新手绘风
- 整体风格：浅色背景、柔和蓝绿/薄荷/米白配色、手绘线条、轻量水彩质感、圆角卡片
- 色调：全篇保持同一套清爽主色，不要每张图风格跳变
- 字体与标注：少量、清晰、统一，适合技术博客阅读
- 信息密度：至少一张图必须做成可截图收藏的对照图或流程图，包含“能力、场景、行动”的对应关系
- 目标：让整篇文章看起来像同一组手绘技术配图，而不是临时拼起来的素材

## 封面图

- 文件名：`封面图-YYYYMMDD-HHMMSS.png`
- 用途：公众号封面 / 文章头图
- 内容：一张能概括全文主题的主视觉，不要塞过多文字
- 风格：清新手绘风，和正文配图保持同一套主色和设计语言
- 比例：优先 16:9，必要时保留中心安全区，方便后续裁切

## 配图1

- 位置：导语后
- 文件名：`配图1-YYYYMMDD-HHMMSS.png`
- 用途：说明“为什么这个主题最近又被重新讨论”
- 内容：一张展示 RAG、长上下文、记忆三者关系的概念图
- 风格：清新手绘风，适合技术博客头图
- 比例：16:9
```

## 配图占位符规则

文章里要主动预留配图位置，不要等用户提醒。

占位符格式固定写成：

```md
[配图1](./assets/配图1-YYYYMMDD-HHMMSS.png)
```

规则：

- `image-requirements.md` 里必须额外包含 1 张封面图，封面图不需要插入正文
- `image-requirements.md` 的“整体要求”必须声明风格预设，默认是 `清新手绘风`
- `image-requirements.md` 的“整体要求”必须声明信息密度要求，至少 1 张正文图要承担总结、对照或行动指南功能
- 短文通常预留 `2` 张图
- 中长文通常预留 `3` 到 `4` 张图
- 图片位置应自然嵌入，不要集中堆在文末
- `blog.md` 里出现的占位符数量，要和 `image-requirements.md` 里的图片需求一一对应
- `image-requirements.md` 开头要先写一段整体要求，保证全篇配图风格一致
- 图片文件名尽量统一携带同一批次时间标签，便于后续发布、去重和素材追踪
- 常见插图位置：
  - 导语后：放趋势图、总览图、示意图
  - 讲原理时：放结构图、流程图
  - 讲对比时：放对比图、表格图
  - 结尾前：放判断框架或行动建议图

## 写作要求

- 先讲“为什么现在值得关注”，再讲“这东西是什么”
- 少写百科式定义，尽量用场景和判断带动阅读
- 标题要优先从读者收益出发，少用只面向作者视角的“盘点 / 介绍 / 总结”
- 如果主题里出现 `最流行`、`最热门`、`Top N`、`最常用`，但素材里没有明确排行榜、下载量、星标、时间范围和排序依据，标题必须降级为读者收益型表达，例如 `最值得先装`、`新手优先装`、`第一批该配置`
- 如果标题或正文保留 `最流行`、`最热门`、`Top N` 等排序表达，必须在开头给出排序口径，并在正文补充排序表；表格要尽量包含下载量、星标、引用热度、更新时间或其他可核验指标
- 不要只堆链接摘要，要形成自己的主线
- 默认吸收素材资料里的观点和事实，不要在正文中途直接写“某篇文章说了什么”或大段贴链接
- 除非用户明确要求，否则不要在文章中间插入“引用来源”式段落；资料更适合转化成自己的分析
- 这类文章不能把本地仓库内容直接当成热门度证据；本地 repo 只能作为示例来源
- 面向中文 / 国内读者时，不要默认把 Google、Gmail、Google Workspace、YouTube、X/Twitter 等作为主路径
- 如果素材里的工具强依赖海外服务，必须明确写出“国内环境的替代路径或使用前提”，例如企业微信 / 飞书 / 钉钉、GitHub / Gitee、网页文章 / B 站 / 本地文档等
- 国内读者向文章应优先回答“在国内能不能用、需要什么前提、有没有替代方案”，不要只写海外工作流
- 允许有明确立场，但不要编造数据、经历或结论
- 对技术读者来说，最有价值的是：
  - 新变化
  - 真问题
  - 真实取舍
  - 下一步怎么判断和实践

## 获取更多读者的写法

- 标题优先写成“趋势 + 判断”或“问题 + 结论”
- 工具推荐类标题优先写成“人群 + 收益 + 行动”，例如 `OpenClaw 新手最值得先装的 5 个 skill`
- 开头前 3 段要尽快回答“这和我有什么关系”
- 正文里尽量给出一个清晰判断，而不是只罗列资料
- 结尾最好落到“谁该现在开始、谁可以先观望”
- 如果有争议点，适合正面点出来，这更容易引发讨论
- 结尾至少给出一个完整 workflow 或第一步行动，让读者知道看完之后怎么照着做

## 与 ai-topic-research 的配合

如果用户还没有主题：

1. 先调用 `ai-topic-research --mode discover`
2. 拿到：
   - 选择主题
   - 支撑链接
   - 写作建议
3. 再围绕“选择主题”调用一次搜索，补充 `素材资料`
4. 最后再用本 skill 生成博客文件夹

## 注意

- 本 skill 输出的是博客文件夹，不负责发布
- 图片先用占位符，不要假装已经有真实素材
- 封面图和正文配图都写进 `image-requirements.md`
- 如果文章涉及“热门 / Top / 最流行”，先确认素材资料里已经有外部社区证据，再开始写
- 如果用户之后要发公众号，可以再交给相关公众号类 skill 处理
