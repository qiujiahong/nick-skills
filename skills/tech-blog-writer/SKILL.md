# Tech Blog Writer

这个 skill 用来把一张“选题卡片”扩写成一个可继续加工的博客文件夹，而不是只输出一段正文。它也是技术博客流水线里的写作中枢：上游承接选题研究和补素材，下游交给 `image-gen` 生成配图，再交给 `wechat-mp-publisher` 渲染、建草稿和发布。

它最适合接在 `ai-topic-research` 后面使用：

- `ai-topic-research` 负责选题、给支撑链接和写作建议
- 再补一轮主题搜索，整理出可用的素材资料
- 本 skill 负责把这些输入整理成一个博客目录，目录里包含正文和配图要求

## 适用场景

- 用户已经有选题，不想从空白页开始写
- 用户已经有几条支撑链接，希望整理成一篇博客
- 用户希望文章自带配图占位符，后续再补图
- 用户想写技术博客，而不是公众号排版稿或营销软文

## 输入要求

正常情况下，尽量具备这三类输入：

- `选择主题`
- `素材资料`
- `写作建议`

如果还有 `支撑链接`，更适合把它作为“补素材”阶段的起点，而不是直接拿来写正文。

如果缺一部分，也可以继续写，但要优先保留已有信息，不要编造事实。

## 端到端标准流程

当用户要完整产出一篇技术博客时，默认按这个顺序执行：

1. 先用 `ai-topic-research` 的 `discover` 模式选题
2. 围绕已选主题再补一轮素材
3. 再调用本 skill 生成博客目录
4. 用 `image-gen` 逐张生成 `assets/` 里的图片
5. 用 `wechat-mp-publisher` 生成公众号风格 HTML、创建草稿，用户确认后再发布

不要跳过中间产物。

原因：

- 选题卡片决定写作方向
- 素材资料决定论据、结构和排序口径
- 博客正文决定配图需求
- 配图要求决定图片 prompt
- 本地渲染 HTML 是公众号发布前的检查点

这是一个严格串联流程，不是几个互相独立的步骤。

### 1. 选题

先调用 `ai-topic-research` 的 `discover` 模式。

输入通常包含：

- 种子方向，例如 `AI编程`
- 最近 10 天已经写过的主题

输出只保留三块：

- 选择主题
- 支撑链接
- 写作建议

### 2. 补素材

不要拿 `discover` 的输出直接开写。

先围绕已选主题再做一轮搜索，补出真正可写作的素材资料。

建议输入：

- 选择主题
- `discover` 阶段给出的支撑链接
- 当前想面向的读者场景，例如 `AI编程`、`Agent`、`企业知识助手`

建议输出：

- `素材资料`
  - 关键事实
  - 关键观点
  - 可用案例
  - 可吸收的对比角度

要求：

- 素材阶段的目标是“给写作补料”，不是直接产出博客
- 尽量把资料整理成笔记式输入，而不是一串裸链接
- 后续写文时默认吸收这些资料，不在正文中途点名引用“某篇文章说”

### 3. 写博客

再调用本 skill。

输入：

- 选择主题
- 素材资料
- 写作建议

输出：

- `blog.md`
- `image-requirements.md`

要求：

- `blog.md` 里必须预留图片占位符，例如 `[配图1](./assets/配图1-YYYYMMDD-HHMMSS.png)`
- `image-requirements.md` 开头必须先写“整体要求”
- `image-requirements.md` 的“整体要求”必须声明配图风格预设，默认选择 `清新手绘风`
- `image-requirements.md` 的“整体要求”必须声明信息密度要求，至少 1 张正文图要做成可截图收藏的对照图、流程图或行动指南图
- `image-requirements.md` 必须额外包含 1 张封面图
- `image-requirements.md` 里的图片编号，必须和 `blog.md` 中占位符一一对应
- 同一篇文章的图片文件名尽量复用同一个时间标签，便于发布时去重和追踪

### 4. 生成配图

再调用 `image-gen`，逐张生成 `assets/` 里的图片文件。

输入来源：

- `image-requirements.md`

输出目标：

- `assets/封面图-YYYYMMDD-HHMMSS.png`
- `assets/配图1-YYYYMMDD-HHMMSS.png`
- `assets/配图2-YYYYMMDD-HHMMSS.png`
- `assets/配图3-YYYYMMDD-HHMMSS.png`

### 5. 发布到公众号

最后调用 `wechat-mp-publisher`。

建议顺序：

1. 先用 `render-content` 生成公众号风格 HTML，本地预览
2. 再用 `create-draft` 创建草稿
3. 用户确认后，才调用 `publish`

封面图和正文图片规则：

- 封面图：上传为公众号素材，拿到 `thumb_media_id`
- 正文图片：上传为公众号正文图片 URL，再替换进 HTML
- `create-draft` 可以直接接受本地 `--thumb-file`，脚本内部会先上传封面图
- Markdown 里的本地图片路径，脚本会在创建草稿前自动上传并替换

## 本 skill 工作流程

1. 先理解主题和读者
2. 从素材资料里提炼文章主线和论据
3. 把写作建议转成标题、开头、结构和收束方式
4. 输出博客文件夹
5. 生成 `blog.md`
6. 生成 `image-requirements.md`
7. 为整篇文章规划 1 张封面图和若干正文配图
8. 在 `blog.md` 合适位置插入配图占位符

## 配图风格预设

生成 `image-requirements.md` 时必须先选择一个统一配图风格。默认使用 `清新手绘风`，除非用户明确指定其他风格。

可用预设：

- `清新手绘风`（默认）：浅色背景、柔和蓝绿/薄荷/米白配色、手绘线条、轻量水彩质感、圆角卡片、少量拟物小图标，适合公众号技术文章，亲和但不幼稚
- `科技信息图风`：深色或渐变背景、发光连线、节点图、仪表盘、科技蓝/青色，适合架构、协议、系统工程主题
- `极简线框风`：白底、细线、少量强调色、结构清楚，适合教程、步骤、概念解释
- `杂志拼贴风`：卡片拼贴、轻微纸张纹理、截图式模块、醒目的标题区，适合趋势判断、工具盘点
- `产品界面风`：接近真实应用 UI 的面板、列表、状态卡、命令行窗口，适合工具使用方法和 workflow 展示

要求：

- `image-requirements.md` 的“整体要求”必须写明选用的风格预设
- 同一篇文章的封面图和正文图必须使用同一个风格预设
- 默认不要用深色赛博风，除非主题确实需要
- 默认不要让图片里出现大量文字，避免生成不准或公众号端阅读吃力
- 对技术工具教程类文章，优先用“清新手绘风 + 轻量流程图/卡片图”的组合
- 配图不能只做氛围装饰，必须至少有 `1` 张高信息密度图，能被读者截图收藏
- 高信息密度图优先表达：`工具/概念 -> 能力 -> 使用场景 -> 第一步行动`
- 即使是清新手绘风，也要把图做成“可读的信息卡片 / 流程图 / 对照图”，而不是只有机器人、云朵、卡片这类泛化元素

## 热门 / Top / 最流行主题规则

如果文章主题里出现以下任何一种表达：

- `最流行`
- `最热门`
- `Top N`
- `最常用`
- `最值得装`
- `爆火`

必须先做一轮外部社区证据搜索，不能直接根据本地仓库内容下结论。

证据优先级：

1. 官方站点 / 官方技能市场 / 官方文档
2. GitHub 仓库活跃度，例如 star、fork、最近更新、issue / discussion 热度
3. 社区文章、教程、帖子、社交讨论
4. 本地仓库和当前工作区

限制：

- 本地工作区内容只能作为“例子”或“已有实现参考”
- 不能把“本地 repo 里有哪些 skill”当成“社区最流行的 skill”
- 如果缺少明确下载榜或官方排行榜，必须在素材阶段先写清楚这篇文章的排序依据

素材阶段至少要补出：

- 排序依据
- 每个入选项的社区证据
- 为什么没选本地仓库里的其他 skill

标题约束：

- 如果素材里没有明确排行榜、下载量、星标、时间范围和排序口径，不要把标题写成 `最流行`
- 从读者角度优先写成 `最值得先装`、`新手优先装`、`第一批该配置`
- 如果标题保留 `最流行 / Top N`，正文开头必须列出排序依据，不能只在文末放链接，并且正文要补排序表
- 排序表要尽量包含下载量、星标、引用热度、更新时间或其他可核验指标；缺少这些指标时，标题必须改成 `最值得先装` 这类读者收益表达

中文 / 国内读者约束：

- 不要默认把 Google、Gmail、Google Workspace、YouTube、X/Twitter 等作为主路径
- 如果入选工具强依赖海外服务，素材和正文都必须写出国内使用前提或替代路径
- 国内替代路径可优先考虑企业微信、飞书、钉钉、Gitee、B 站、网页文章、本地文档、内部知识库等

## 默认输出目录

正常情况下，输出一个目录，路径约定为：

```text
tech-blog-writer/YYYYMMDD/<topic-slug>/
  blog.md
  image-requirements.md
  assets/
```

例如：

- `tech-blog-writer/20260410/rag-long-context-memory/`
- `tech-blog-writer/20260410/agent-protocols/`
- `tech-blog-writer/20260410/ai-coding-agents/`

### `blog.md`

这是正文文件，正常情况下包含：

1. 标题
2. 导语
3. 正文
4. 结尾

正文内部建议包含：

- 为什么这个主题现在值得写
- 它解决了什么问题
- 社区现在在怎么讨论它
- 读者应该怎么判断要不要跟进
- 读者看完以后第一步应该怎么做

### `image-requirements.md`

这是配图需求文件，正常情况下每张图都要写清楚：

- 整体风格要求
- 封面图要求
- 图片编号
- 建议放置位置
- 图片用途
- 画面内容
- 风格建议
- 尺寸 / 比例建议

建议格式像这样：

```md
## 整体要求

- 风格预设：清新手绘风
- 整体风格：浅色背景、柔和蓝绿/薄荷/米白配色、手绘线条、轻量水彩质感、圆角卡片
- 色调：全篇保持同一套清爽主色，不要每张图风格跳变
- 字体与标注：少量、清晰、统一，适合技术博客阅读
- 信息密度：至少一张图必须做成可截图收藏的对照图或流程图，包含“能力、场景、行动”的对应关系
- 目标：让整篇文章看起来像同一组手绘技术配图，而不是临时拼起来的素材

## 封面图

- 文件名：`封面图-YYYYMMDD-HHMMSS.png`
- 用途：公众号封面 / 文章头图
- 内容：一张能概括全文主题的主视觉，不要塞过多文字
- 风格：清新手绘风，和正文配图保持同一套主色和设计语言
- 比例：优先 16:9，必要时保留中心安全区，方便后续裁切

## 配图1

- 位置：导语后
- 文件名：`配图1-YYYYMMDD-HHMMSS.png`
- 用途：说明“为什么这个主题最近又被重新讨论”
- 内容：一张展示 RAG、长上下文、记忆三者关系的概念图
- 风格：清新手绘风，适合技术博客头图
- 比例：16:9
```

## 配图占位符规则

文章里要主动预留配图位置，不要等用户提醒。

占位符格式固定写成：

```md
[配图1](./assets/配图1-YYYYMMDD-HHMMSS.png)
```

规则：

- `image-requirements.md` 里必须额外包含 1 张封面图，封面图不需要插入正文
- `image-requirements.md` 的“整体要求”必须声明风格预设，默认是 `清新手绘风`
- `image-requirements.md` 的“整体要求”必须声明信息密度要求，至少 1 张正文图要承担总结、对照或行动指南功能
- 短文通常预留 `2` 张图
- 中长文通常预留 `3` 到 `4` 张图
- 图片位置应自然嵌入，不要集中堆在文末
- `blog.md` 里出现的占位符数量，要和 `image-requirements.md` 里的图片需求一一对应
- `image-requirements.md` 开头要先写一段整体要求，保证全篇配图风格一致
- 图片文件名尽量统一携带同一批次时间标签，便于后续发布、去重和素材追踪
- 常见插图位置：
  - 导语后：放趋势图、总览图、示意图
  - 讲原理时：放结构图、流程图
  - 讲对比时：放对比图、表格图
  - 结尾前：放判断框架或行动建议图

## 图片 prompt 组织规则

所有图片 prompt 都要先继承 `image-requirements.md` 里的“整体要求”。

也就是每张图都应该保持：

- 同一套主色系
- 同类构图风格
- 同类信息图语言
- 同类字体与标注感觉

再叠加每张图自己的：

- 文件名
- 用途
- 内容
- 比例
- 放置位置

## 图片生成并发规则

非常重要：

- 一次最多只允许 `2` 个并发图片生成任务
- 不允许一次性把 3 张或 4 张图同时发给上游

推荐执行方式：

1. 先启动 2 张
2. 任意 1 张完成后，再补下一张
3. 始终保持并发数 `<= 2`

这样做的原因：

- 上游更容易限流
- 同时跑太多张时更容易出现 `429`
- 串行补位更容易排查失败图片

## 图片生成失败处理

如果出现以下情况：

- `429`
- upstream overloaded
- 长时间无返回
- 输出文件未落盘

处理规则：

1. 不要继续增加新的并发任务
2. 记录失败的是哪一张图
3. 对失败图片单独重试
4. 重试时优先降低压力：
   - 保持单张重试
   - 必要时降低 `image-size`
   - 必要时简化 prompt

建议：

- 某张图连续失败时，先让其他已成功的图保留，不要整批重跑
- 对比图、结构图这类复杂信息图，更容易触发慢请求，应该优先单独重试

## 写作要求

- 先讲“为什么现在值得关注”，再讲“这东西是什么”
- 少写百科式定义，尽量用场景和判断带动阅读
- 标题要优先从读者收益出发，少用只面向作者视角的“盘点 / 介绍 / 总结”
- 如果主题里出现 `最流行`、`最热门`、`Top N`、`最常用`，但素材里没有明确排行榜、下载量、星标、时间范围和排序依据，标题必须降级为读者收益型表达，例如 `最值得先装`、`新手优先装`、`第一批该配置`
- 如果标题或正文保留 `最流行`、`最热门`、`Top N` 等排序表达，必须在开头给出排序口径，并在正文补充排序表；表格要尽量包含下载量、星标、引用热度、更新时间或其他可核验指标
- 不要只堆链接摘要，要形成自己的主线
- 默认吸收素材资料里的观点和事实，不要在正文中途直接写“某篇文章说了什么”或大段贴链接
- 除非用户明确要求，否则不要在文章中间插入“引用来源”式段落；资料更适合转化成自己的分析
- 这类文章不能把本地仓库内容直接当成热门度证据；本地 repo 只能作为示例来源
- 面向中文 / 国内读者时，不要默认把 Google、Gmail、Google Workspace、YouTube、X/Twitter 等作为主路径
- 如果素材里的工具强依赖海外服务，必须明确写出“国内环境的替代路径或使用前提”，例如企业微信 / 飞书 / 钉钉、GitHub / Gitee、网页文章 / B 站 / 本地文档等
- 国内读者向文章应优先回答“在国内能不能用、需要什么前提、有没有替代方案”，不要只写海外工作流
- 允许有明确立场，但不要编造数据、经历或结论
- 对技术读者来说，最有价值的是：
  - 新变化
  - 真问题
  - 真实取舍
  - 下一步怎么判断和实践

## 获取更多读者的写法

- 标题优先写成“趋势 + 判断”或“问题 + 结论”
- 工具推荐类标题优先写成“人群 + 收益 + 行动”，例如 `OpenClaw 新手最值得先装的 5 个 skill`
- 开头前 3 段要尽快回答“这和我有什么关系”
- 正文里尽量给出一个清晰判断，而不是只罗列资料
- 结尾最好落到“谁该现在开始、谁可以先观望”
- 如果有争议点，适合正面点出来，这更容易引发讨论
- 结尾至少给出一个完整 workflow 或第一步行动，让读者知道看完之后怎么照着做

## 与 ai-topic-research 的配合

如果用户还没有主题：

1. 先调用 `ai-topic-research --mode discover`
2. 拿到：
   - 选择主题
   - 支撑链接
   - 写作建议
3. 再围绕“选择主题”调用一次搜索，补充 `素材资料`
4. 最后再用本 skill 生成博客文件夹

## 产出校验

本 skill 结束前，必须检查：

- `blog.md` 存在
- `image-requirements.md` 存在
- `assets/` 存在
