# Tech Blog Writer

这个 skill 用来把一张“选题卡片”扩写成一篇可继续加工的技术博客初稿。

它最适合接在 `ai-topic-research` 后面使用：

- `ai-topic-research` 负责选题、给支撑链接和写作建议
- 本 skill 负责把这些输入写成一篇结构完整的技术博客

## 适用场景

- 用户已经有选题，不想从空白页开始写
- 用户已经有几条支撑链接，希望整理成一篇博客
- 用户希望文章自带配图占位符，后续再补图
- 用户想写技术博客，而不是公众号排版稿或营销软文

## 输入要求

正常情况下，尽量具备这三类输入：

- `选择主题`
- `支撑链接`
- `写作建议`

如果缺一部分，也可以继续写，但要优先保留已有信息，不要编造事实。

## 工作流程

1. 先理解主题和读者
2. 从支撑链接里提炼文章主线
3. 把写作建议转成标题、开头、结构和收束方式
4. 输出完整博客初稿
5. 在合适位置插入配图占位符

## 默认输出结构

正常情况下，按这个顺序输出：

1. 标题
2. 导语
3. 正文
4. 结尾

正文内部建议包含：

- 为什么这个主题现在值得写
- 它解决了什么问题
- 社区现在在怎么讨论它
- 读者应该怎么判断要不要跟进

## 配图占位符规则

文章里要主动预留配图位置，不要等用户提醒。

占位符格式固定写成：

```md
[配图1](./assets/配图1.png)
```

规则：

- 短文通常预留 `2` 张图
- 中长文通常预留 `3` 到 `4` 张图
- 图片位置应自然嵌入，不要集中堆在文末
- 常见插图位置：
  - 导语后：放趋势图、总览图、示意图
  - 讲原理时：放结构图、流程图
  - 讲对比时：放对比图、表格图
  - 结尾前：放判断框架或行动建议图

## 写作要求

- 先讲“为什么现在值得关注”，再讲“这东西是什么”
- 少写百科式定义，尽量用场景和判断带动阅读
- 不要只堆链接摘要，要形成自己的主线
- 允许有明确立场，但不要编造数据、经历或结论
- 对技术读者来说，最有价值的是：
  - 新变化
  - 真问题
  - 真实取舍
  - 下一步怎么判断和实践

## 获取更多读者的写法

- 标题优先写成“趋势 + 判断”或“问题 + 结论”
- 开头前 3 段要尽快回答“这和我有什么关系”
- 正文里尽量给出一个清晰判断，而不是只罗列资料
- 结尾最好落到“谁该现在开始、谁可以先观望”
- 如果有争议点，适合正面点出来，这更容易引发讨论

## 与 ai-topic-research 的配合

如果用户还没有主题：

1. 先调用 `ai-topic-research --mode discover`
2. 拿到：
   - 选择主题
   - 支撑链接
   - 写作建议
3. 再用本 skill 写成博客初稿

## 注意

- 本 skill 输出的是博客初稿，不负责发布
- 图片先用占位符，不要假装已经有真实素材
- 如果用户之后要发公众号，可以再交给相关公众号类 skill 处理
